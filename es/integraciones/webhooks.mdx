Los webhooks son notificaciones HTTP que Recurrente envía automáticamente a tu servidor cada vez que ocurre un evento importante en tu cuenta, como un pago completado, un pago fallido o un cambio en el estado de una suscripción. En lugar de tener que consultar la API periódicamente, tu servidor recibe la información en tiempo real y puede reaccionar de inmediato.

## Cómo configurar un webhook

<Steps>
  <Step title="Accede a la sección de Webhooks">
    Ingresa a [app.recurrente.com](https://app.recurrente.com) y ve a **Mi Cuenta → Desarrolladores y API**. Haz clic en el botón **Webhooks**. Serás redirigido automáticamente al panel de Svix, la plataforma que Recurrente usa para gestionar la entrega confiable de webhooks.
  </Step>

  <Step title="Agrega un endpoint">
    En el panel de Svix, haz clic en **Add Endpoint** (esquina superior derecha). En el campo **Endpoint URL**, ingresa la URL de tu servidor que recibirá los eventos:

    ```
    https://tudominio.com/webhooks/recurrente
    ```

    La URL debe ser pública y accesible desde internet. Asegúrate de que responda con un código HTTP `200` al recibir el payload.
  </Step>

  <Step title="Selecciona los eventos (opcional)">
    Por defecto, el endpoint recibirá todos los eventos. Si lo prefieres, puedes filtrar los tipos de eventos específicos que quieres recibir usando la configuración en Svix.
  </Step>

  <Step title="Guarda y prueba el endpoint">
    Haz clic en **Create** para guardar el endpoint. Desde la pestaña **Testing** del panel de Svix puedes enviar un ejemplo de cualquier evento del catálogo a tu endpoint y ver el payload, los intentos de entrega y si fue exitosa.
  </Step>
</Steps>

## Probar tu integración

Hay dos formas de generar webhooks sin mover dinero real:

- **Pagar un checkout de prueba con la tarjeta `4242 4242 4242 4242`.** Se disparan `payment_intent.succeeded` e `intent.succeeded` (`type: payment`) únicamente a los endpoints registrados con llaves **test** (`pk_test_*` / `sk_test_*`). Los pagos de prueba solo simulan tarjeta.
- **Usar la pestaña Testing del panel de Svix.** Envía a tu endpoint un ejemplo de cualquier evento del catálogo, incluso los que no puedes reproducir con un pago de prueba (suscripciones, disputas, transferencias, etc.).

## Eventos disponibles

Recurrente emite eventos para pagos, suscripciones, reembolsos, disputas y movimientos de dinero. Los más comunes:

### Eventos de pago (`intent.*`)

| Evento | Cuándo se emite |
|---|---|
| `intent.succeeded` | Cobro exitoso con cualquier método de pago. Los fondos ya están en tu balance. |
| `intent.pending` | Cobro iniciado o pendiente (transferencia, cripto). |
| `intent.failed` | Cobro fallido. |
| `intent.canceled` | Intent cancelado. |
| `intent.paid` | Solo para pagos con balance: llega a la cuenta pagadora si tiene un endpoint configurado. |

### Eventos de suscripción

| Evento | Cuándo se emite |
|---|---|
| `subscription.create` | Se crea una suscripción. |
| `subscription.update` | Cambia información general de la suscripción, como el método de pago. |
| `subscription.past_due` | El cobro automático falla por primera vez. |
| `subscription.pause` | Se pausa la suscripción. No se vuelve a cobrar hasta reactivarla. |
| `subscription.unpause` | Se reanuda una suscripción pausada. |
| `subscription.reactivate` | Se reactiva una suscripción cancelada dentro del período permitido. |
| `subscription.cancel` | El cobro automático falla por tercera vez y la suscripción se cancela. |
| `subscription.item_added`, `subscription.item_removed` | Se agrega o remueve un producto recurrente. |
| `subscription.invoice_item_added` | Se agrega un cargo único a la próxima factura. |

### Otros eventos

| Evento | Cuándo se emite |
|---|---|
| `setup_intent.succeeded` | Se inicia una suscripción con período de prueba o se tokeniza una tarjeta sin cobrarla. |
| `setup_intent.cancelled` | No se logra tokenizar una tarjeta (p. ej. el primer pago de una suscripción con prueba falla). |
| `refund.create` | Reembolso procesado. |
| `dispute.create`, `dispute.update` | Eventos de contracargo. |
| `transfer.sent`, `transfer.received` | Transferencia P2P enviada o recibida. |
| `withdrawal.create`, `withdrawal.update` | Retiro bancario creado o cambio de estado. |
| `swap.create`, `swap.update` | Envío de stablecoin creado o cambio de estado. |
| `payment_intent.*`, `bank_transfer_intent.*`, `crypto_intent.*`, `balance_intent.*`, `cash_intent.*` | Eventos legacy por método de pago. Siguen funcionando sin cambios. |

<Note>
Puedes ver el catálogo completo, el esquema y un payload de ejemplo de cada evento en la [referencia de la API](https://docs.recurrente.com/) o desde la pestaña **Event Catalog** del panel de Svix.
</Note>

## Ejemplo de payload

Cuando Recurrente envía un evento a tu endpoint, el body de la solicitud tiene una estructura JSON similar a esta:

```json
{
  "event_type": "payment_intent.succeeded",
  "created_at": "2025-03-15T14:32:00Z",
  "data": {
    "id": "pi_xxxxxxxxxxxxxxxx",
    "amount": 15000,
    "currency": "GTQ",
    "status": "succeeded",
    "customer_email": "cliente@ejemplo.com",
    "checkout_id": "ch_xxxxxxxxxxxxxxxx",
    "metadata": {}
  }
}
```

Usa el campo `event_type` para determinar qué acción tomar en tu lógica de negocio.

## Buenas prácticas

<Accordion title="Verifica la firma del webhook">
  Svix firma cada solicitud con un secreto único por endpoint. Verifica siempre la firma antes de procesar el evento para asegurarte de que la notificación proviene genuinamente de Recurrente y no de un tercero. Puedes encontrar el secreto de firma en la configuración del endpoint dentro del panel de Svix.
</Accordion>

<Accordion title="Usa siempre HTTPS">
  Tu endpoint debe estar protegido con HTTPS. Los endpoints con HTTP plano no son seguros para recibir datos de pago y pueden ser rechazados por Svix.
</Accordion>

<Accordion title="Responde con 200 lo antes posible">
  Tu servidor debe responder con un código HTTP `200` en menos de unos pocos segundos. Si la respuesta tarda demasiado o devuelve un error, Svix intentará reenviar el evento automáticamente según su política de reintentos. Procesa la lógica pesada de forma asíncrona (en una cola de trabajos) y devuelve el `200` de inmediato.
</Accordion>

<Accordion title="Maneja entregas duplicadas">
  En casos raros, un mismo evento puede entregarse más de una vez. Diseña tu lógica para que sea idempotente: si recibes el mismo `id` de evento dos veces, el resultado debe ser el mismo que si lo hubieras recibido solo una vez.
</Accordion>
